## 1. 제품 개요

장바구니에 담긴 상품의 **소계·할인·최종 결제 금액**을 계산하는 시스템이다.

- **Entity** (`src/cart.py`): 순수 도메인 로직. Flask 등 Boundary를 import 하지 않는다.
- **Boundary** (`src/app.py`): Flask 주문 폼. Entity를 import 해 재사용한다.

모든 동작은 **계약 ID**(INV-*, E-*)로 추적한다. **ID에 없는 동작은 구현하지 않는다.**

---

## 2. 발견 근거 (MomTest)

본 PRD의 요구사항은 기능 명세가 아니라, 아래 **실제 사례**에서 추출한 테스트 가능한 계약이다.

### 2.1 인터뷰 메모

- **2024-03-12**: A 3개(12,000원) + B 1개(30,000원) = 66,000원. 5만 원 넘으면 10% 할인. 기대 59,400원.
- **VIP**: 문턱 할인 후 추가 5% 할인. 적용 순서는 **문턱 → VIP** 고정.

### 2.2 실제 주문 사례

| 주문 | 소계 | 결제 | 비고 |
|------|------|------|------|
| #1107 | 60,000 | 54,000 | 10% 할인 적용 |
| #1042 | 48,000 | 48,000 | 할인 없음 |

### 2.3 엑셀 검산

- 66,000 × 0.9 = 59,400 (`ROUND`)
- VIP 추가: 59,400 × 0.95 = 56,430

### 2.4 CS 클레임 / 버그

- 수량 `-1` 입력 시 화면 합계 **0원** 표시 → 기대: **오류 또는 거부**
- `items` 없이 제출 시 **HTTP 500** → 기대: 검증 오류 (500 금지)

### 2.5 명시적 비범위 / 미확인

- 쿠폰·배송비: 언급 없음 (범위 외)
- 소계 **정확히 50,000원**: "넘으면"만 확인, 포함 여부 **미확인**

## 3. 계약 정의

### 3.1 불변식 (Invariant) — Entity

| ID | 계약 | 근거 레벨 | 검증 위치 |
|----|------|-----------|-----------|
| **INV-001** | `calculate_subtotal(price, quantity)`는 `price * quantity`를 반환한다. | L1 | `tests/entity/` |
| **INV-002** | 장바구니 소계는 모든 라인의 `price * quantity` 합과 같다. | L1 | `tests/entity/` |
| **INV-003** | 소계 `> 50_000`이면 `round(subtotal * 0.9)`를 반환한다. 소계 `≤ 50_000`이면 소계를 그대로 반환한다. | L1 | `tests/entity/` |
| **INV-004** | VIP이면 INV-003 결과에 `round(amount * 0.95)`를 적용한다. 비VIP이면 INV-003 결과를 그대로 쓴다. **적용 순서: 문턱(10%) → VIP(5%) 고정.** | L2 | `tests/entity/` |
| **INV-005** | 유효 입력·정의된 할인 규칙 하에서 최종 결제 금액은 소계를 초과하지 않는다. | L3 | `tests/entity/` |

#### 검증 데이터 (INV-003, INV-004)

| 입력 소계 | VIP | 기대 결제 | 근거 |
|-----------|-----|-----------|------|
| 48,000 | — | 48,000 | #1042 |
| 60,000 | — | 54,000 | #1107 |
| 66,000 | No | 59,400 | 인터뷰 2024-03-12 |
| 66,000 | Yes | 56,430 | 엑셀 검산 |

> **INV-003 경계 해석:** 인터뷰 표현 "5만 원 **넘으면**"과 #1042(48,000 미할인)·#1107(60,000 할인)에 따라 **`> 50_000`만 계약에 포함**한다. `== 50_000`은 [OOS-003](#32-범위-외-oos) 참조.

### 3.2 에러 계약 (Error Contract)

| ID | 계약 | 근거 레벨 | 검증 위치 |
|----|------|-----------|-----------|
| **E-001** | `quantity < 0`이면 `ValueError`를 발생시킨다(인덱스 포함). **0원으로 조용히 계산하지 않는다.** | L0 | `tests/entity/` |
| **E-002** | `items`가 없거나 빈 목록이면 HTTP 500이 아닌 검증 오류(4xx 등)를 반환한다. | L0 | `tests/boundary/` |

### 3.3 범위 외 (OOS)

| OOS ID | 만들지 않을 동작 | 제외 이유 |
|--------|------------------|-----------|
| **OOS-001** | 쿠폰 할인 | 인터뷰 명시적 비범위 |
| **OOS-002** | 배송비 | 인터뷰 명시적 비범위 |
| **OOS-003** | 소계 **정확히 50,000원** 시 할인 여부 | 포함 여부 미확인 |
| **OOS-004** | VIP + 소계 ≤ 50,000 (문턱 미적용 후 VIP 5%) | 해당 주문 사례 없음 |
| **OOS-005** | `price < 0` 처리 | CS 사례는 수량 -1만 있음 |
| **OOS-006** | `quantity == 0` 처리 | 사례·클레임 없음 |
| **OOS-007** | 포인트·회원 등급(VIP 외) | 근거 없음 |
| **OOS-008** | 할인 적용 순서 변경(VIP → 문턱) | 인터뷰 "문턱 → VIP 고정"만 확인 |
| **OOS-009** | 소수 할인율·1원 미만 `ROUND` 세부 규칙 | 제공 사례는 모두 정수; 소수 발생 사례 없음 |

## 4. 미확인 갭 (Discovery 잔여 질문)

아래에 대한 **사건 근거**가 확보되기 전까지 계약·테스트에 반영하지 않는다.

1. 소계가 **정확히 50,000원**이었던 주문 — 할인 적용 여부?
2. **VIP**인데 소계가 50,000원 이하(예: 48,000원)인 주문 — 결제 금액?
3. 할인 결과에 **1원 미만 소수**가 생긴 주문 — `ROUND` 자릿수·방식?
4. **단가 음수**·**수량 0** 실제 입력 사례와 기대 처리?
5. `items` 없이 제출 시 사용자에게 보여줄 **구체적 오류 메시지**?

## 5. 구현 범위 및 TDD 순서

워크플로: **RED → GREEN → REFACTOR** ([AGENTS.md](../AGENTS.md))

| 순서 | 계약 ID | RED 대상 | 비고 |
|------|---------|----------|------|
| 1 | INV-001 | `tests/entity/test_cart.py` | RED 테스트 존재 |
| 2 | INV-002 | `tests/entity/` | 장바구니 합산 |
| 3 | INV-003 | `tests/entity/` | 문턱 10% 할인 |
| 4 | INV-004 | `tests/entity/` | VIP 5% (문턱 후) |
| 5 | INV-005 | `tests/entity/` | 결제 ≤ 소계 |
| 6 | E-001 | `tests/entity/` | 음수 수량 |
| 7 | E-002 | `tests/boundary/` | `src/app.py` 필요 |

### 테스트 명령

```bash
pytest -q                  # 전체
pytest tests/entity -q     # Entity (INV-*)
pytest tests/boundary -q   # Boundary (E-*)
```

### 구현 규칙

- GREEN 단계: 실패를 통과시키는 **최소 구현**만. 구현 줄에 계약 ID 주석 (`# INV-001`).
- RED 단계: `tests/` 만 수정. `src/` 는 건드리지 않는다.
- 커밋 분리: `test`(RED) / `feat`(GREEN) / `refactor` — 메시지에 계약 ID 포함.
