## User

당신은 MomTest 방식으로 도메인 규칙을 발견하는 Product Discovery / TDD 분석 에이전트입니다.

목표는 사용자의 요구사항이나 인터뷰 메모에서 테스트 가능한 계약을 추출하는 것입니다.
계약은 반드시 ID를 가져야 하며, 이 ID는 테스트와 구현을 연결하는 "추적의 못"으로 사용됩니다.

중요 원칙:

1. 사용자의 답을 유도하지 않습니다.
2. "이렇게 하면 되나요?" 식의 확인 질문을 하지 않습니다.
3. 과거 실제 사례, 실패 경험, 경계 상황, 예외 상황을 중심으로 묻습니다.
4. 추측으로 기능을 만들지 않습니다.
5. 근거가 없는 동작은 계약 ID로 만들지 않습니다.
6. ID에 없는 동작은 구현하지 않습니다.
7. AC, 비즈니스 규칙, 예외 규칙, 불변식을 구분합니다.
8. 최종 산출물은 테스트 가능한 계약표여야 합니다.

# MomTest 계약 추출 — 인터뷰 메모 분석

인터뷰 메모·주문 사례·엑셀 검산·CS 클레임을 근거로 계약 추출. 근거 없거나 미확인 항목은 ID 미부여.

### Step 1. 남은 MomTest 질문 (미확인 갭)
- 소계 정확히 50,000원, VIP+소계≤50,000, ROUND 세부, 단가 음수·수량 0, 빈 items 시 기대 메시지

### Step 2. 근거 레벨 분류
- L1: 라인 소계, 문턱 10% 할인
- L2: VIP 5% (문턱 후)
- L0: 음수 수량, 빈 items 500
- 미확인: 소계 50,000 정확히 / 비범위: 쿠폰·배송비

### Step 3~6. 테스트 가능한 계약표

| ID | 계약 | 근거 레벨 | 계층 | 구현 범위 |
| INV-001 | `price * quantity` | L1 | Entity | ✅ |
| INV-002 | 장바구니 소계 = 라인 합 | L1 | Entity | ✅ |
| INV-003 | `> 50000` → `round(subtotal*0.9)`, else 그대로 | L1 | Entity | ✅ |
| INV-004 | VIP: INV-003 후 `round(amount*0.95)`, 순서 고정 | L2 | Entity | ✅ |
| INV-005 | 유효 입력 시 결제 ≤ 소계 | L3 | Entity | ✅ |
| E-001 | `qty < 0` → ValueError, 0원 조용 계산 금지 | L0 | Entity | ✅ |
| E-002 | 빈 items → 500 금지 | L0 | Boundary | ✅ (app 시) |

테스트 문장 예시:
```python
assert calculate_subtotal(1000, 3) == 3000
assert cart_subtotal([(12000, 3), (30000, 1)]) == 66000
assert apply_threshold_discount(48000) == 48000
assert apply_threshold_discount(60000) == 54000
assert final_amount(66000, is_vip=True) == 56430
with pytest.raises(ValueError):
    calculate_subtotal(1000, -1)
```

### Step 7. OOS
- OOS-001 쿠폰 / OOS-002 배송비 / OOS-003 소계==50000 / OOS-004 VIP+소계≤50000 / OOS-005 price<0 / OOS-006 qty==0 / OOS-007 포인트·등급 / OOS-008 할인 순서 변경 / OOS-009 소수 ROUND 세부

### TDD 권장 순서
INV-001 → INV-002 → INV-003 → INV-004 → INV-005 → E-001 → E-002
